Prepare for The Merge

New user?

This guidance is targeted at users who are already running Agora-cl. If you're starting from scratch, see our Quickstart.

Merge preparation checklist#

Agora-cl v3.1.1 is a Merge-ready release. Review the release notes to understand how this release impacts your configuration.

If you've ever set the USE_PRYSM_VERSION environment variable to use a release candidate, either clear it via UNSET USE_PRYSM_VERSION (Linux/MacOS) or set USE_PRYSM_VERSION= (Windows).

Verify that you're running Agora-cl v3.1.1 by issuing the following command: agora-cl.sh beacon-chain --version (Linux) agora-cl.bat beacon-chain --version (Windows).

The Before and now section below gives you a high-level overview of the items that you need to keep in mind while preparing for The Merge. Merge readiness checklist for more detailed information.

Review the post-Merge system requirements section below to ensure that your configuration will support The Merge. Note that a 2TB+ SSD is highly recommended.

  • Execution client:
  • Agora
Use agora-el version to check the version. See Agora-el's releases page and join Telegram to stay up to date as we approach Mainnet Merge.

  • EN-BN connection:
  • HTTP-JWT
  • IPC

If you're not using IPC to connect your Agora node and execution node, ensure that both your execution node and Agora node are configured to use JWT authentication. These instructions are included below, and are also available here: Configure JWT

If you're not using IPC to connect your Agora node and execution node, your Agora node will need to connect to its execution node on port 8551. Previously, port 8545 was used. If your Agora node and execution node are on different host machines, ensure that your firewall rules are updated accordingly, and refer to Configure ports and firewalls for improved network connectivity for general connectivity improvement tips.

If you're running a validator, configuring a fee recipient address will allow you to earn what were previously miners' transaction fee tips. Instructions are provided below, and also here: Configure a Fee Recipient address.

See Check node and validator status to learn how to check the status of your execution node, Agora node, and validator node.

Best practices#

  • If you're staking BOA as a validator, try this guide on a testnet first, then mainnet.
  • Keep things simple. This guidance assumes all client software will run on a single machine.
  • Review the latest advisories for the network(s) that you're using: testnet, Mainnet.
  • Review all of our published security best practices.
  • Join the community - join the Bosagora Telegram for updates and support.

Configure JWT authentication

The HTTP connection between your Agora node and execution node needs to be authenticated using a JWT token. There are several ways to generate this JWT token:

  • Use an online generator like this. Copy and paste this value into a jwt.hex file.
  • Use a utility like OpenSSL to create the token via command: openssl rand -hex 32 | tr -d "\n" > "jwt.hex".
  • Use an execution client to generate the jwt.hex file.
  • Use Agora-cl to generate the jwt.hex file:
  • Operating system:
  • Linux, MacOS, Arm64
  • Windows
## Optional. This command is necessary only if you've previously configured USE_PRYSM_VERSIONSET USE_PRYSM_VERSION=v3.1.1
## Required.agora-cl.bat beacon-chain generate-auth-secret
## Optional. This command is necessary only if you've previously configured USE_PRYSM_VERSIONUSE_PRYSM_VERSION=v3.1.1
## Required../agora-cl.sh beacon-chain generate-auth-secret

Agora-cl will output a jwt.hex file path.

Ensure that the script, user, or terminal window used to create and access your JWT token has the permissions it needs. Windows users may need to run command windows as Administrator.

Configure execution node#

Your execution node needs to expose a new port and then use the JWT token to authenticate your Agora node's connection to that port.

Using the latest version of your execution client software, issue the following command to configure your execution node's JWT token and Engine API endpoint:

  • Agora-el

Download and run the latest 64-bit stable release of the Agora-el installer for your operating system from the Agora execution layer client downloads page.

Navigate to your execution directory and run the following command to start your execution node:

  • Mainnet
  • Testnet
  • JWT
  • IPC
geth --http --http.api eth,net,engine,admin --authrpc.jwtsecret /path/to/jwt.hex 
geth --http --http.api eth,net,engine,admin
  • JWT
  • IPC
geth --goerli --http --http.api eth,net,engine,admin --authrpc.jwtsecret /path/to/jwt.hex 
geth --goerli --http --http.api eth,net,engine,admin

See Geth's command-line options for parameter definitions.

Syncing can take a long time - from hours to days. You can proceed to the next step while your execution node syncs.

Configure Agora node#

Next, we'll configure your Agora node to consume your JWT token so it can form an authenticated HTTP connection with your execution node.

In this step, you'll run a Agora node using Agora-cl.

  • Windows
  • Linux, MacOS, Arm64
  • Mainnet
  • Testnet

Use the following command to start a Agora node that connects to your local execution node:

  • JWT
  • IPC
agora-cl.bat beacon-chain --execution-endpoint=http://localhost:8551 --jwt-secret=path/to/jwt.hex --suggested-fee-recipient=0x01234567722E6b0000012BFEBf6177F1D2e9758D9

--http-web3provider is deprecated and has been replaced with --execution-endpoint, but IPC currently only works through --http-web3provider on Windows. This will be fixed in our next release. You can safely ignore any related "deprecated flag" warnings you see in the meantime.

agora-cl.bat beacon-chain --http-web3provider=//./pipe/<your.ipc> --suggested-fee-recipient=0x01234567722E6b0000012BFEBf6177F1D2e9758D9

Download the Testnet genesis state into your consensus/agora-cl directory. Then use the following command to start a Agora node that connects to your local execution node:

Configure validator node#

Verify that your Agora node and execution node are both fully synced. If you're not fully synced, you risk being penalized and losing some of your staked BOA.

Other than ensuring that you're using the latest stable Agora-cl release, validator client configuration doesn't need to be updated for The Merge. A fee recipient address can optionally be configured on your validator node if you want redundancy or multiple fee recipient addresses. See Configure a Fee Recipient address to learn more.

Ensure that you're not running multiple instances of the same validator public key, especially if you're using scripts or other forms of automation. If the Ethereum network detects two instances of the same validator key submitting proposals, attestations, or votes, it may assume malicious intent and slash accordingly.

Frequently asked questions#

I'm currently running a validator on Agora Mainnet. When should I make changes?
You can now make these changes, regardless of the network you're running on.

Can I use IPC post-Merge?
Yes. You also won't have to worry about JWT if you use IPC. See our Quickstart for IPC instructions.

Can I use a light node with Agora-cl, or do I need to run a full execution node?
No - at this time, a full node is required.
